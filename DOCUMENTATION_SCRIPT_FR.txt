================================================================================
              LOW_LATENCY_GAMING_SCRIPT.bat  --  DOCUMENTATION
                  Fonctionnement complet, detection, ordre
================================================================================

Ce document explique de A a Z comment le script principal fonctionne : sa
structure, comment il s'adapte a ta machine, et dans quel ordre il applique
chaque reglage. Tout est en francais.


--------------------------------------------------------------------------------
1. CE QUE FAIT LE SCRIPT (et sa philosophie)
--------------------------------------------------------------------------------

C'est un optimiseur "low latency" pour Windows 11, oriente jeu competitif. Il
reduit le lag d'entree (input lag) et stabilise la reactivite, SANS casser les
anticheats et SANS toucher a la securite materielle (Secure Boot, TPM, VBS /
Memory Integrity, isolation du noyau restent ON ; le Defender reste actif).

Principe : il applique un grand nombre de reglages connus (registre, services,
plan d'alimentation, bcdedit, taches planifiees), regroupes en 47 sections
numerotees [0] a [46]. Beaucoup de reglages sont CONDITIONNELS : ils s'adaptent
a ton GPU, a ton CPU (notamment X3D), et au mode choisi.


--------------------------------------------------------------------------------
2. STRUCTURE DU FICHIER
--------------------------------------------------------------------------------

- C'est un .bat pur (aucun .exe, aucune dependance externe).
- Encodage : Windows-1252 (ANSI), fins de ligne CRLF. A garder tel quel.
  -> Si tu le re-sauvegardes, choisis ANSI / Windows-1252, JAMAIS UTF-8,
     sinon les accents des commentaires et des libelles casseraient.
- Il commence par "@echo off" + "chcp 1252" et se termine par "exit /b".
- Il s'auto-eleve en administrateur (UAC) au lancement.
- Il utilise "setlocal EnableDelayedExpansion" : les variables se lisent avec
  !VAR! (et pas %VAR%), ce qui permet de les modifier en cours d'execution.

Organisation interne du code :
  a) Bloc commun de tete (elevation, couleurs ANSI, detection de la langue).
  b) Bloc des LIBELLES en anglais (les variables de texte M0, M4, ...),
     qui se termine par l'etiquette  :_strdone.
  c) Bloc des LIBELLES en francais (etiquette :_strfr ... :_strdone).
  d) Apres :_strdone : tout le code PARTAGE (menu, detection materielle,
     application des 47 sections, resume, et le mode restauration).
  -> Seuls les TEXTES different entre EN et FR. La logique, les couleurs et la
     mise en page sont identiques dans les deux langues.


--------------------------------------------------------------------------------
3. DEROULEMENT A L'EXECUTION (etape par etape)
--------------------------------------------------------------------------------

  1) Auto-elevation : si pas admin, le script se relance en admin et quitte.
  2) Definition des couleurs ANSI (vert, cyan, jaune, rouge, orange, bleu...).
  3) Detection de la langue du systeme -> choisit les libelles EN ou FR.
  4) Banniere d'accueil (titre + sous-titre + signature).
  5) MENU : le script demande
        - la langue (si besoin),
        - le MODE : "soft" (equilibre) ou "lowest" (latence minimale),
        - puis applique, ou bascule en mode restauration.
  6) DETECTION MATERIELLE (voir section 4 ci-dessous).
  7) APPLICATION des sections [0] a [46], dans l'ordre, chacune affichant
     une ligne de confirmation  [ OK ] [NN] <description>.
  8) RESUME final : recapitulatif de ce qui a ete applique.
  9) Invitation a REDEMARRER (certains reglages ne sont pleinement actifs
     qu'apres reboot : parking selon CPU, demande timer globale, MSI mode, etc.).


--------------------------------------------------------------------------------
4. LA DETECTION MATERIELLE (comment le script "voit" ta machine)
--------------------------------------------------------------------------------

Juste avant d'appliquer, un bloc DETECTION interroge Windows via PowerShell
(Get-CimInstance) et remplit des variables :

  GPUVENDOR  -> "NVIDIA", "AMD" ou "OTHER"
               (lu depuis le nom de la carte video Win32_VideoController)
  RAMGB      -> quantite de RAM en Go (arrondie)
  CPUVENDOR  -> "Intel", "AMD" ou "OTHER" (depuis Win32_Processor.Name)
  ISX3D      -> "1" si le nom du CPU contient "X3D", sinon "0"
                -> l'ecran de detection au demarrage affiche "AMD X3D mono-CCD"
                   (ou "AMD X3D bi-CCD" si le CPU est multi-CCD)
  CPUCORES   -> nombre de coeurs physiques
  X3DMULTI   -> "1" si CPU AMD X3D MULTI-CCD (X3D + plus de 8 coeurs, ex 7950X3D),
                sinon "0"
  ISLAPTOP   -> "1" si portable (batterie presente OU chassis de type portable),
                sinon "0"
  HASWIFI    -> "1" si une carte Wi-Fi physique est presente, sinon "0"

Ces variables servent ensuite a CHOISIR les bons reglages :
  - GPUVENDOR aiguille la section [04] (reglages NVIDIA OU AMD).
  - ISX3D aiguille le parking + les C-states de la section [36] (voir section 8).
  - X3DMULTI aiguille tscsyncpolicy [07] (active seulement sur AMD X3D bi-CCD).
  - ISLAPTOP / HASWIFI aiguillent les reglages portables (voir section 5,
    sous-section PORTABLE).
  - RAMGB calcule AUTOMATIQUEMENT le seuil svchost [01] :
        SvcHostSplitThresholdInKB = RAM en Ko (repli 32 Go).
    Resultat : Windows REGROUPE les services en moins de processus svchost
    (au lieu d'un processus par service au-dela de 3,5 Go). Tweak intelligent,
    adapte a la RAM detectee.
  - CPUCORES sert d'information / de garde-fou.


--------------------------------------------------------------------------------
5. LES DEUX MODES : "soft" vs "lowest"
--------------------------------------------------------------------------------

Au menu, tu choisis un mode. La majorite des sections sont identiques ; voici
TOUTES les differences entre les deux modes :

  - SystemResponsiveness (reserve multimedia) :
        soft = 10                      |   lowest = 0
  - Win32PrioritySeparation (quantums) :
        soft = 38                      |   lowest = 42
  - disabledynamictick (tick fixe) :
        soft = no                      |   lowest = yes
  - Plan d'alimentation :
        soft = Performances elevees    |   lowest = Performances optimales (Ultimate)
  - IFEO, priorite par jeu (section [03]) :
        soft = defaut (aucune)         |   lowest = priorite haute
  - TdrDelay, timeout GPU (section [04]) :
        soft = defaut (supprime)       |   lowest = 8 secondes

Dans les DEUX modes, la surcouche parking / C-states adaptee au CPU (section 8)
est appliquee PAR-DESSUS le plan choisi (elle reste prioritaire).

"lowest" pousse la latence plus loin (priorites jeu, timeout GPU rallonge, tick
fixe, plan Ultimate) au prix d'une conso/chaleur superieure. "soft" reste
l'equilibre : pas de priorite par jeu, timeouts par defaut, plan Performances
elevees, CPU plus frais.

Le timer 0.5 ms (section [45]) est INDEPENDANT du mode : applique pareil dans
les deux cas, et concu pour tenir 24/7 (voir section 9).


  PORTABLE (laptop) : AUTODETECTION ET REGLAGES PRIORITAIRES
  ----------------------------------------------------------
  Le script detecte automatiquement si la machine est un portable (batterie
  presente + chassis de type portable) ou un PC fixe (variable ISLAPTOP).

  Sur un PC FIXE : rien ne change, tu choisis soft ou lowest normalement, et
  tout ce qui est decrit plus haut s'applique tel quel.

  Sur un PORTABLE : tu choisis TOUJOURS soft ou lowest, MAIS 6 reglages sont
  FORCES (prioritaires sur le mode choisi) pour preserver batterie et thermique.
  Ils ecrasent les valeurs du mode pour ces 6 points precis uniquement :

    1. Timer 0.5 ms uniquement sur secteur (le timer est power-aware, voir
       section 9 ; sur batterie il relache).
    2. Eco Wi-Fi seulement sur batterie, et seulement si une carte Wi-Fi est
       presente (secteur = perf max, batterie = economie max). Sinon on n'y
       touche pas.
    3. ULPS gardee ACTIVE pour les GPU AMD (sur fixe, lowest la desactive).
    4. Parking CPU restaure par defaut + C-states gardes (priorite au CPU, au
       lieu du deparkage force du mode fixe).
    5. Plan d'alimentation force en Performances elevees (jamais Ultimate).
    6. disabledynamictick force en no (tick fixe non applique).

  Le reste du mode choisi (soft ou lowest) s'applique normalement.

  PERSISTANCE : la tache de re-application [46] (qui restaure les reglages
  apres une MAJ majeure de Windows) est CONSCIENTE du portable. Elle lit
  IsLaptop en registre et re-applique les 6 overrides cote portable (timer
  power-aware, eco Wi-Fi si carte presente, ULPS gardee, parking par defaut,
  plan Performances elevees, dynamictick no). Les 6 reglages portables
  PERSISTENT donc apres une grosse MAJ Windows, SANS relancer le script.


--------------------------------------------------------------------------------
6. L'ORDRE DES TWEAKS  --  les 47 sections [0] a [46]
--------------------------------------------------------------------------------

L'ordre suit une logique : d'abord la performance brute, puis le debloat et la
vie privee, puis les peripheriques, puis l'interface, puis le systeme, et enfin
la maintenance. Detail :

  PERFORMANCE / LATENCE
   [0]  Elevation administrateur
   [01] Noyau CPU / RAM (seuil svchost selon RAM, priorites, timer global, SysMain)
   [02] MMCSS (priorites multimedia pour les jeux)
   [03] IFEO : priorite des executables de jeu
   [04] GPU (NVIDIA ou AMD, dont ULPS off cote AMD) + MSI mode + TdrDelay (lowest)
   [05] Souris + clavier + accessibilite + anti pop-up Store
   [06] Reseau + QoS (RSS on, RSC off, file QoS...)
   [07] Timers bcdedit + NTFS (tick fixe selon mode ; tscsync si AMD X3D bi-CCD)

  CONFIDENTIALITE / NETTOYAGE
   [08] Telemetrie        [09] Taches de telemetrie
   [09b] Hosts : blocage pub + telemetrie MS (215 domaines) + exclusion Defender
   [10] Services          [11] Indexation
   [12] FSO (Fullscreen Optimizations)
   [13] Game DVR + Game Mode
   [14] Exclusions Defender (les jeux, pour eviter le scan en pleine partie)
   [15] Debloat (apps inutiles)   [16] Vie privee
   [17] Camera

  PERIPHERIQUES
   [18] Wifi / Ethernet   [19] Bluetooth   [20] Imprimante
   [21] Xbox              [22] OneDrive

  INTERFACE / CONFORT
   [23] Edge                  [24] Apparence   [25] Barre des taches
   [26] Menu Demarrer         [27] Explorateur [28] Eclairage dynamique
   [29] Multitache            [30] UAC         [31] Bureau a distance

  SYSTEME
   [32] Memory Integrity (laisse ON)   [33] Boot
   [34] Peripheriques fantomes         [35] FTH (Fault Tolerant Heap)
   [36] Alimentation (plan + parking + C-states adaptatifs : voir section 8)
   [37] HAGS (verifie le support PUIS active)
   [38] Demarrage rapide (laisse ON)  [39] Icones de demarrage
   [40] Pilotes

  MAINTENANCE
   [41] Optimisation disque   [42] Storage Sense   [43] Nettoyage
   [44] Windows Update (Microsoft Update, Y COMPRIS les MAJ facultatives) + winget
   [45] Timer 0.5 ms (resolution + tache au demarrage)
   [46] Re-application au demarrage (voir section 9)

Chaque section affiche  [ OK ] [NN] <description>  quand elle est passee.


--------------------------------------------------------------------------------
7. LES TWEAKS CONDITIONNELS (comment ca s'adapte)
--------------------------------------------------------------------------------

  GPU (section [04]) :
    - si GPUVENDOR = NVIDIA  -> reglages NVIDIA (dont EnableGR535 = 0)
    - si GPUVENDOR = AMD     -> reglages AMD, dont EnableUlps = 0 (empeche le GPU
      de tomber en etat ultra-basse conso -> evite des micro-coupures)
    - le MSI mode est applique pour tous ; le TdrDelay uniquement en mode lowest.

  MODE (soft / lowest) :
    - change plusieurs valeurs : reserve multimedia, quantums, tick fixe, plan
      d'alimentation, priorite par jeu (IFEO) et TdrDelay (voir section 5).

  CPU / X3D :
    - parking + C-states selon ISX3D (section [36], voir section 8) ;
    - tscsyncpolicy enhanced seulement si AMD X3D bi-CCD (X3DMULTI, section [07]).

Tout le reste est applique de maniere identique sur toutes les machines.


--------------------------------------------------------------------------------
8. PARKING + C-STATES ADAPTATIFS SELON LE CPU (section [36])
--------------------------------------------------------------------------------

C'est la partie la plus "intelligente". Le script choisit DEUX comportements
opposes selon que le CPU est un X3D ou non :

  CPU X3D (ex : 7800X3D, 9800X3D) :
    - Parking : LAISSE ACTIF (CPMINCORES = 0). Au repos, des coeurs se parquent
      pour laisser un coeur booster plus haut -> meilleur single-core.
    - C-states : ACTIVES (idle autorise). L'X3D chauffe et throttle : le laisser
      respirer reduit le stutter.

  CPU NON-X3D ou INTEL :
    - Parking : DESACTIVE (unpark, CPMINCORES = 100). Tous les coeurs restent
      prets -> reactivite maximale.
    - C-states : DESACTIVES (idle off). Pas de latence de reveil de coeur.

Comment c'est rendu PRIORITAIRE sur le plan d'alimentation :
  Le plan est d'abord active (Performances optimales / Ultimate en lowest,
  Performances elevees en soft). Or ces plans forcent eux-memes l'unpark. Le
  script REECRIT ENSUITE les valeurs CPMINCORES et IDLEDISABLE (AC + DC) sur le
  plan ACTIF, PUIS re-active le plan. Comme cette reecriture a lieu APRES
  l'activation, ce sont les valeurs adaptees au CPU qui gagnent -- quel que soit
  le mode. Le pilote chipset AMD + un reboot finalisent le parking reel.

Note honnete : sur un X3D MONO-CCD (8 coeurs sur un seul CCD, cas du 7800X3D),
le gain du parking sur le single-core est MODESTE. Le gros benefice du parking
concerne surtout les puces BI-CCD (type 7950X3D). Cote non-X3D, desactiver les
C-states augmente la chaleur/conso pour un gain de latence marginal : c'est le
compromis "low latency classique", assume.


--------------------------------------------------------------------------------
9. TIMER 0.5 ms [45] ET RE-APPLICATION AU DEMARRAGE [46]
--------------------------------------------------------------------------------

  [45] Timer 0.5 ms (concu pour tenir 24/7) :
    - pose GlobalTimerResolutionRequests = 1 (rend la demande GLOBALE, pas juste
      pour le process courant),
    - ecrit un petit script PowerShell (timerres.ps1 dans C:\Windows) qui appelle
      en boucle NtSetTimerResolution(5000) (0.5 ms), protege par un mutex.
      POWER-AWARE : il lit l'etat d'alimentation et n'applique le 0.5 ms que sur
      secteur ; sur batterie il relache (PC fixe = continu, portable = secteur
      uniquement),
    - cree une tache planifiee ROBUSTE "TimerRes 0.5ms" :
        * 2 declencheurs : AU DEMARRAGE (AtStartup, compte SYSTEM, AVANT meme
          l'ouverture de session) ET a CHAQUE OUVERTURE DE SESSION (AtLogOn) ;
        * privileges eleves, sans limite de duree ;
        * REDEMARRAGE AUTO si le processus s'arrete (3 essais, 1 min).
      -> le timer est actif des le boot, system-wide, et se relance tout seul.
    - le lance immediatement.
    (Le script separe TIMER_0.5MS.bat installe la MEME tache robuste.)

  [46] Re-application au demarrage :
    - cree une tache "LowLat Reapply" qui, A CHAQUE DEMARRAGE, verifie si une
      MAJ Windows a change le build du systeme. Si OUI seulement, elle RE-POSE
      l'INTEGRALITE des reglages susceptibles de sauter apres une MAJ, selon le
      mode, le materiel ET l'etat portable (IsLaptop) enregistres a l'apply : reserve multimedia +
      NetworkThrottling, Win32PrioritySeparation, disabledynamictick, MMCSS jeux,
      timer global, PowerThrottling, GameDVR off / Game Mode on, DirectX swap,
      WSearch, vie privee, souris (precision off), HAGS, SvcHostSplitThreshold,
      tscsyncpolicy (selon bi-CCD), TdrDelay (selon mode), priorites par jeu IFEO
      (selon mode), NVIDIA EnableGR535 et mode MSI du GPU.
    - Si le build n'a pas change, elle saute cette partie registre.
    - A CHAQUE demarrage (meme sans MAJ), elle effectue les routines SURES :
      * GARANTIT le timer 0.5ms : si la tache "TimerRes 0.5ms" a disparu (ex
        apres une grosse MAJ) elle la RECREE + remet GlobalTimerResolutionRequests
        = 1  -> timer bulletproof aux updates.
      * RE-ACTIVE le plan d'alim choisi (Ultimate / Hautes perf) + RE-POSE le
        parking / C-states adaptes au CPU (depuis la detection stockee). Le plan
        d'alim peut retomber sur Equilibre apres une veille ou une MAJ : on le
        garantit donc a chaque boot (logique X3D respectee).
      * SUPPRIME les peripheriques fantomes (Status 'Unknown') via pnputil.
      * vide les dossiers Temp (utilisateur + Windows) et fait ipconfig /flushdns.
    - Les decisions materielles (X3D, bi-CCD, RAM->Ko) sont stockees a l'apply
      dans HKLM\SOFTWARE\LowLatOptimizer pour relecture sans re-detection.


--------------------------------------------------------------------------------
10. LE MODE RESTAURATION
--------------------------------------------------------------------------------

Depuis le menu, on peut lancer la restauration. Elle remet Windows dans un etat
propre : valeurs par defaut du planificateur et du reseau, suppression des
taches planifiees creees, retour au plan d'alimentation "Equilibre" (qui parque
nativement et reactive les C-states), retrait des reglages recents (anti pop-up
Store, TdrDelay), et remise des mitigations Spectre/Meltdown au defaut Windows
(activees). Les exclusions Defender et le debloat ne sont pas tous annules
automatiquement (sans danger a laisser).
